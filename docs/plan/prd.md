## 1. 問題定義

### v1 で解決済みの課題

UI リデザインにより、明るくポップな配色、グラスモーフィズム、ボタンの視認性向上、正解時のフィードバック強化は完了している。基本的なアニメーション（数字ボタンのポップイン、スコアのバウンス、紙吹雪エフェクト、ボタンタップのスケール変化）も実装済み。

### v2 で解決済みの課題

1. アニメーションの物足りなさ
   - 背景のアンビエントアニメーション、式入力時の演出、正解時の多彩な祝福エフェクトにより改善済み

2. 行き詰まり時の救済手段がない
   - ギブアップ機能（答え参照 + 次の問題へ）により改善済み

### v5 で取り組む課題

Level 1（超かんたん）と Level 2（かんたん）の難易度がまだ高い

- Level 1 の穴埋めは空欄が1箇所のみのため、ほぼ「選ぶだけ」で終わってしまう。手応えがなさすぎて退屈に感じる一方で、Level 2 への移行時に急に難しくなる段差がある
- Level 2 はヒント付きとはいえ、括弧を使う解法しか存在しない組み合わせが出題されることがある。括弧の概念を理解していない子どもにとっては、ヒントがあっても解けない壁になる
- Level 1 から Level 2、Level 2 から Level 3 への難易度の階段が急すぎ、レベルアップ直後に「全然できない」と感じて離脱するリスクがある

### v3 で取り組む課題（実装中）

初めて遊ぶ子どもに対する導入体験がない

- 現在、ゲーム開始直後から「4つの数字と演算子で式を組み立てる」フル操作を要求する。7歳の子どもにとって、白紙の状態から式全体を構築するのはハードルが高い
- 四則演算をまだ完全には理解していない段階の子どもは、最初の問題で「何をすればいいか分からない」まま止まってしまう
- 初回プレイで成功体験を得られないと、アプリを閉じて二度と開かない
- 段階的に難しくなる仕組みがないため、既に慣れた子にも新鮮さの変化がない
- 現在のスコア（星の数）は「何問正解したか」を示すだけで、成長実感やマイルストーンとしての意味がない

### v4 で取り組む課題

ゲームの「顔」がなく、第一印象が弱い

- アプリを開いた瞬間にゲーム画面（問題）が表示される。何のアプリか、どう遊ぶのかの導入がなく、いきなり始まる感覚がある
- favicon がデフォルトの Vite アイコンのまま残っており、ブラウザのタブや PWA のホーム画面アイコンがゲームの内容を表現していない
- PWA アイコン（icon-192.png, icon-512.png）もプレースホルダー状態で、ホーム画面に追加したときの見栄えが悪い
- 「Make 10」というゲーム名やブランドを印象づけるビジュアル要素が存在しない
- 再訪時やホーム画面からの起動時に「このアプリは何だったか」を思い出す手がかりがない

## 2. ターゲットユーザー

### ペルソナ A: 小学校低学年の子ども（7〜9歳）

- 足し算・引き算を学び始めた時期
- スマートフォンやタブレットに触り慣れている
- カラフルで動きのあるものに惹かれる
- 正解できたときの「やった!」感が遊び続けるモチベーション
- 分からないと飽きて別のアプリに移ってしまう
- (v3 追加) 最初はルールや操作方法がよく分からず、手が止まりやすい
- (v3 追加) 「自分でもできた!」という最初の成功体験が、継続プレイへの分岐点になる
- (v4 追加) アプリを開いたときに「ゲームが始まるぞ!」というワクワク感があると、遊ぶ気持ちのスイッチが入る
- (v5 追加) 穴埋めが1箇所だけだと簡単すぎて飽きるが、4つ全部を自分で組み立てるのはまだ難しい。2箇所埋める形式がちょうどいい手応えになる
- (v5 追加) 括弧の意味がまだ分からないので、括弧が必要な問題が出ると「どうやっても合わない」と感じてしまう

### ペルソナ B: 子どもに算数を楽しんでほしい保護者

- 子どもに渡して安心できるシンプルなアプリを求めている
- ゲームとしての楽しさと学習効果の両立を期待
- 広告や課金要素のない、PWA としてインストールできるものを好む
- 子どもが諦めずに続けられる仕組みがあると嬉しい
- (v3 追加) 段階的に力がつく学習設計だと安心感がある
- (v3 追加) 子どもの成長が目に見える仕組み（レベルアップ）を喜ぶ
- (v4 追加) ホーム画面のアイコンがかわいく、子ども向けだと一目で分かると安心して渡せる
- (v5 追加) レベルアップ直後に「急に難しすぎる」と子どもが投げ出さないよう、段階が滑らかだと安心できる

## 3. バリュープロポジション

v1 の「見た瞬間に遊びたくなる」体験と v2 の「分からなくても大丈夫」体験に加え、v3 では以下の価値を追加する:

- 初めての子どもでも「できた!」から始まる導入体験。最初は穴埋め形式で1箇所だけ答えればよく、ゲームのルールと成功体験を同時に学べる。星を集めるうちに自然とフル操作に到達する、スムーズなスキルの階段。

v4 では以下の価値を追加する:

- アプリを開いた瞬間から「楽しそう!」と感じるゲーム開始画面。タイトルロゴとスタートボタンが、遊ぶ前の期待感を高める。ブラウザのタブやホーム画面のアイコンもゲームの世界観を反映し、「あのゲームだ」と一目で分かる。

v5 では以下の価値を追加する:

- Level 1 の空欄を2箇所にすることで、「選ぶだけ」から「考えて選ぶ」への小さなステップが生まれる。1箇所目を埋めて、2箇所目も埋めるという順序的な思考が、Level 2 以降の式組み立てへの橋渡しになる。Level 2 では括弧なしで解ける組み合わせだけを出題することで、四則演算の基本に集中できる。括弧という追加の概念を排除し、「数字と演算子を並べれば10が作れる」成功体験を積み重ねてから Level 3 に進む、なだらかな難易度カーブを実現する。

### Must Have（v4 で必須）

- M15: ゲーム開始画面（タイトルスクリーン）

  アプリ起動時にゲーム開始画面を全画面で表示する。ゲーム本体はこの画面を抜けた後に表示する。

  - ゲームタイトル「Make 10」を画面中央上部に大きく表示する
  - タイトルの下にゲームの簡単な説明文を1行で表示する（例: 「4つの すうじで 10を つくろう!」）
  - 画面中央に「スタート」ボタンを配置する。ボタンは目立つデザインにし、タップを誘導する
  - 背景は既存のアンビエントアニメーション付きグラデーションを流用し、ゲーム本体との世界観を統一する
  - 「スタート」をタップするとゲーム開始画面がフェードアウトし、ゲーム本体に遷移する
  - 2回目以降のアクセスでも毎回開始画面を表示する（スキップ機能は設けない）

- M16: favicon の刷新

  ブラウザタブに表示される favicon を、ゲームの内容を表現するキャッチーなデザインに差し替える。

  - 「10」の数字をモチーフにした、カラフルで視認性の高いデザインにする
  - 既存のカラーパレット（amber, pink, orange 系の暖色）と調和するカラーリングにする
  - SVG 形式で作成し、小さいサイズ（16x16px, 32x32px）でも判別できるシンプルさを保つ
  - index.html の favicon 参照を新しいファイルに差し替える

- M17: PWA アイコンの刷新

  ホーム画面追加時やアプリ一覧に表示される PWA アイコンを、ゲームの世界観に合った本格的なデザインに差し替える。

  - favicon と同一のモチーフをベースに、大きいサイズ向けにディテールを追加する
  - 192x192px と 512x512px の2サイズを PNG 形式で用意する
  - 丸角の背景にタイトルやモチーフを配置し、ホーム画面で他のアプリと並んでも見劣りしないクオリティにする
  - maskable アイコン（セーフゾーン内にコンテンツを収めた版）も用意し、Android のアダプティブアイコンに対応する
  - vite.config.ts の PWA manifest 設定で新しいアイコンファイルを参照する

### Must Have（v5 で必須）

- M18: Level 1（超かんたん）の空欄を2箇所に変更

  現在の Level 1 は正解式の中から1箇所だけを空欄にする穴埋め形式だが、これを2箇所に増やす。

  - 正解式の中から、数字または演算子の位置を2箇所ランダムに空欄にする
  - 2箇所の空欄は順番に埋める。1箇所目の空欄をタップして選択肢から選び、正解したら2箇所目の空欄に進む
  - 1箇所目の空欄を正しく埋めるまで、2箇所目には進めない（順序制御）
  - 各空欄に対してそれぞれ独立した選択肢（3〜4個）を提示する
  - 2箇所とも正解した時点でスコアが +1 され、正解フィードバックが表示される
  - 不正解の場合は該当の空欄で間違いフィードバックを表示し、同じ空欄の選択肢からやり直す（従来と同じ挙動）
  - ギブアップ時は2箇所とも正解を表示する

- M19: Level 2（かんたん）の出題を括弧なし限定に変更

  現在の Level 2 は solvableCombinations からランダムに出題するが、括弧を使わずに10が作れる組み合わせのみに限定する。

  - 出題する4つの数字の組み合わせは、少なくとも1つの括弧なし正解式（数字と演算子のみの式）が存在するものに限定する
  - 括弧なし正解式とは、式の中に `(` と `)` を含まない解法のこと（例: `1+2+3+4` は括弧なし、`(1+2)×3+1` は括弧あり）
  - 出題時の組み合わせ選択ロジック（generatePuzzle 相当）でフィルタリングする。solver で解を求めた後、括弧なしの解が1つも存在しない組み合わせは除外して再生成する
  - ヒント表示や判定ロジックなど、Level 2 のその他の挙動は変更しない
  - 判定自体は従来どおり式全体で行う。ユーザーが括弧を使って正解した場合も正解として扱う（括弧の使用を禁止するわけではない）

### Must Have（v3 で必須）

- M10: 3段階の難易度レベル

  ゲームは以下の3段階で進行する。各レベルは星の獲得数によって自動的に昇格する。

  | レベル | 名前 | 星の範囲 | 内容 |
  |--------|------|----------|------|
  | 1 | 超かんたん | 0〜4 | 穴埋め形式: 完成済みの正解式の中から、演算子または数字が2箇所空欄になっている。ユーザーは空欄を順番にタップして、選択肢から正しいものを選ぶ (v5 で1箇所→2箇所に変更) |
  | 2 | かんたん | 5〜9 | 部分組み立て形式: 括弧なしで解ける4つの数字が出題される。2つはヒント表示され、残り2つの数字と演算子をユーザーが埋める (v5 で括弧なし限定に変更) |
  | 3 | ふつう | 10〜 | 現在と同じフル操作: 4つの数字から式全体を自由に組み立てる |

- M11: 「超かんたん」モードの出題ロジック

  - solver が算出した正解式を1つ選び、その中の演算子または数字を2箇所空欄（ブランク）にする (v5 で1箇所→2箇所に変更)
  - 空欄の位置はランダムに決定する
  - 各空欄に対する選択肢を3〜4個提示する（正解1つ + ダミー2〜3つ）
  - ダミーの選択肢は、見た目にもっともらしいが計算結果が10にならないものにする
  - ユーザーは空欄を順番にタップして選択肢から選ぶ。2箇所とも正解すると回答完了 (v5 で順序制御を追加)

- M12: 「かんたん」モードの出題ロジック

  - 括弧なしで解ける4つの数字の組み合わせのみを出題対象とする (v5 で追加)
  - solver の正解式から、4つの数字のうち2つを式の中にあらかじめ配置した状態で出題する
  - 配置済みの2つの数字はボタンが「使用済み」状態で表示される
  - 残り2つの数字と必要な演算子はユーザーが自由に入力する
  - 判定ロジックはふつうモードと同じ（式全体が10になるかを評価する。括弧を使った正解も受け付ける）

- M13: 難易度レベルの自動昇格

  - スコア（星）の累計に基づいて、レベルの切り替えが自動的に発生する
  - レベル昇格時にお祝い演出を表示する（「レベルアップ!」+ レベル名 + 短い祝福アニメーション）
  - 昇格は一方向のみ。レベルが下がることはない
  - スコアとレベルは localStorage に永続化する

- M14: 現在のレベル表示

  - ヘッダーエリアに現在の難易度レベル名を表示する（「超かんたん」「かんたん」「ふつう」）
  - 次のレベルまでの残りスター数がユーザーに伝わる表示にする（例: プログレスバー、「あと N もん!」テキスト）

## 5. デザイン方針

### カラーパレット

（v1 から変更なし。実装済みの配色を維持する。）

- 背景: 明るい暖色系グラデーション（amber-100, pink-100, indigo-100）
- 数字ボタン: 4色ポップカラー（rose, sky, emerald, amber）
- 演算子ボタン: violet-400
- コントロールボタン: slate-200, red-100
- 判定ボタン（=）: orange-to-pink グラデーション

### レイアウト

- 既存のレイアウト構造（ヘッダー / 式表示 / 数字パッド / 演算子パッド / コントロールパッド）は維持する
- ギブアップボタンはコントロールパッドの一部として配置する（レイアウト大幅変更なし）
- 背景アンビエント要素は position: fixed で配置し、既存コンテンツを覆わない

### v4 追加: ゲーム開始画面のデザイン

- 全画面のオーバーレイとして実装する（ルーティングは追加しない）
- 背景: ゲーム本体と同じグラデーション + アンビエントアニメーションを使用する
- タイトル「Make 10」: Nunito フォント、太字（800）、大きめ（3rem 程度）、グラデーション文字色または影付きで目立たせる
- サブタイトル: ひらがなの説明文、子どもが読めるサイズ（1rem 程度）
- スタートボタン: 既存の判定ボタン（=）と同系統の orange-to-pink グラデーション、角丸、大きめサイズ、影付き
- 画面下部にさりげなく「Make 10 Puzzle」のクレジットまたはバージョン表記を小さく配置する

### v4 追加: favicon / アイコンのデザイン方針

- モチーフ: 「10」の数字を中心に据える。「1」と「0」をポップなカラーで表現する
- カラー: 既存パレットの暖色系（amber, orange, pink）をベースにする
- 形状: 角丸の四角形背景に数字を配置する。シンプルで小さくても視認できるデザインにする
- PWA アイコンではタイトル「Make 10」の文字を添えてもよい（favicon では省略する）
- maskable アイコンではセーフゾーン（中央 80% の円形エリア）にコンテンツを収める

### v3 追加: 難易度別の UI バリエーション

- 「超かんたん」モード
  - 式表示エリアに正解式を表示し、2箇所の空欄部分をハイライトされたタップ可能なスロットとして表現する (v5 で1箇所→2箇所に変更)
  - 現在回答中の空欄スロットは点線の枠 + パルスアニメーションで「ここをタップしてね」と誘導する
  - 次に回答する空欄スロットは薄い点線の枠で表示し、まだ回答の番ではないことを視覚的に伝える (v5 追加)
  - 1箇所目を正解すると、その空欄に正解が埋まり、2箇所目の空欄がアクティブになる (v5 追加)
  - スロットをタップすると、選択肢が下部にポップアップで表示される（現在のキーパッド位置を流用）
  - 選択肢は大きめのボタン（3〜4つ）で表示し、選びやすくする
  - 通常のキーパッド（数字パッド・演算子パッド）は非表示にする

- 「かんたん」モード
  - 式表示エリアには数字が2つ配置された状態のテンプレートを表示する
  - 配置済みの2つの数字は NumberPad 上で「使用済み」状態にする
  - 残りの操作は「ふつう」モードと同一のキーパッド UI で行う

- 「ふつう」モード
  - 現在の UI そのまま。変更なし

### レベル表示のトーン

- レベル名は子どもが読めるひらがな表記にする（「ちょうかんたん」「かんたん」「ふつう」）
- プログレス表示は前向きなトーン（「あと 3もんで レベルアップ!」）
- レベルアップ時の演出は派手に祝う（「やったね! レベルアップ!」）

### ギブアップ UI のトーン

（v2 から変更なし。）

- ネガティブな表現（「あきらめる」「降参」）は避ける
- 「こたえを みる」のようなフレンドリーな文言にする
- 確認ダイアログも励ましのトーン（「こたえを みても いいかな?」）
- 答え表示画面では「こうやって とくんだね!」のような学びにつなげるメッセージ

### アニメーション原則

（v2 から変更なし。v3 の新規アニメーションにも同じ原則を適用する。）

- 楽しさを生むが邪魔にならない: アンビエント要素は低い不透明度で緩やかに動く
- レスポンシブなフィードバック: ユーザー操作には即座に視覚的反応を返す（100ms 以内）
- 一貫したイージング: ease-out を基本とし、バウンス系は cubic-bezier で統一する
- 過度にしない: 画面上で同時に激しく動く要素は2つまでに制限する

## 6. 非機能要件

### パフォーマンス

- First Contentful Paint: 1.5 秒以内
- アニメーションは 60fps を維持する
- CSS アニメーション / Tailwind のトランジションを優先し、JS アニメーションライブラリの追加は最小限にする
- アンビエントアニメーションは will-change / transform のみを使い、レイアウト再計算を発生させない
- 追加する Web フォントは 1 ファミリーまでに制限する（Nunito 導入済み）
- 答え生成ロジックの計算は出題時に完了させ、ギブアップタップ時にはゼロ遅延で表示する
- (v3 追加) 穴埋め問題の生成（正解式からの空欄作成 + ダミー選択肢の生成）は出題時に完了させる
- (v4 追加) 開始画面の表示は First Contentful Paint に含まれるため、1.5 秒以内を維持する
- (v4 追加) 開始画面からゲーム画面への遷移は 300ms 以内に完了する
- (v5 追加) Level 2 の括弧なし組み合わせフィルタリングによる出題生成が体感遅延なく完了する（再生成のリトライが発生しても 100ms 以内）

### アクセシビリティ

- ボタンの最小タップ領域: 48x48px
- テキストと背景のコントラスト比: WCAG AA 準拠（4.5:1 以上）
- 使用済みボタンの状態が色以外でも判別できる（透明度の変化 + disabled 属性）
- ギブアップボタンに適切な aria-label を付与する
- 答え表示は aria-live="polite" で読み上げ対応する
- `prefers-reduced-motion: reduce` ですべてのアニメーションを無効化する（既存実装を拡張）
- (v3 追加) 穴埋めの空欄スロットに適切な aria-label（「えらんでね」など）を付与する
- (v3 追加) 選択肢ボタンにも aria-label を付与する
- (v5 追加) Level 1 の2箇所空欄で、現在アクティブな空欄と未回答の空欄の区別が色以外でも判別できる（アクティブ空欄のパルスアニメーション + aria-current 等）
- (v4 追加) 開始画面のスタートボタンに適切な aria-label を付与する
- (v4 追加) favicon の SVG にアクセシブルな title 要素を含める

### ブラウザ / デバイスサポート

- iOS Safari 16+
- Android Chrome 最新版
- 画面幅 320px〜428px を主な対象とする
- PWA としてのオフライン動作を維持する

### データ / プライバシー

- ローカルストレージにスコアとレベルを保存する（既存のスコア保存を拡張）
- 外部へのデータ送信は行わない
- Web フォントの読み込み以外に外部リクエストを追加しない

## 7. スコープ境界

v5 の変更が及ぶ範囲:

- src/logic/generateFillInBlank.ts -- 空欄を2箇所に拡張。FillInBlankPuzzle の型定義を複数空欄対応に変更（blankIndex → blankIndices 配列化、choices を空欄ごとに分離）
- src/hooks/useMake10.ts -- Level 1 の selectChoice ロジックを順序制御付き2段階回答に変更。現在の空欄インデックスの状態管理を追加
- src/logic/generatePartialPuzzle.ts -- 出題時に solver の解をフィルタし、括弧なし解が存在する組み合わせのみを採用するロジックを追加
- src/components/ -- Level 1 の穴埋め UI を2箇所空欄対応に更新（現在の空欄ハイライト、次の空欄への遷移表示）

v4 の変更が及ぶ範囲:

- src/components/ -- TitleScreen コンポーネントの新規追加
- src/App.tsx -- 開始画面の表示状態管理と、ゲーム本体への遷移制御を追加
- public/favicon.svg -- 新規作成（キャッチーなゲーム用 favicon）
- public/icons/icon-192.png, icon-512.png -- 実際の PNG 画像に差し替え
- public/icons/icon-maskable-192.png, icon-maskable-512.png -- maskable アイコンの新規追加
- index.html -- favicon の参照先を vite.svg から favicon.svg に変更
- vite.config.ts -- PWA manifest の icons 配列に maskable アイコンを追加、theme_color / background_color をゲームの配色に合わせて調整

v3 の変更が及ぶ範囲（変更なし）:

- src/hooks/useMake10.ts -- 難易度レベル状態の追加、レベル別の出題ロジック切り替え、レベルアップ判定
- src/logic/ -- 穴埋め問題生成ロジック（正解式からの空欄生成、ダミー選択肢生成）の新規追加

以下は明確にスコープ外とする:

- 4つの数字で10を作るゲームルール自体の変更
- solvableCombinations データセットの変更
- スコア加算ロジックの変更（正解 +1 のみ。ギブアップではスコア変動なし。レベルによるスコア倍率もなし）
- 新しいページやルーティングの追加（開始画面は React の条件分岐で実装し、react-router 等は導入しない）
- 外部ライブラリの大規模な追加

## 8. 成功指標

定性的な指標で判断する:

- (v2) 画面を眺めているだけでも楽しいと感じられること
- (v2) 操作のたびに「気持ちいい」と感じられるレスポンスがあること
- (v2) ギブアップが「恥ずかしい」ではなく「勉強になった」と感じられるトーンであること
- (v3 追加) 初めてアプリを開いた子どもが、操作説明なしに最初の問題を正解できること
- (v3 追加) 穴埋め形式からフル操作への移行が自然で、子どもが「急に難しくなった」と感じないこと
- (v3 追加) レベルアップの瞬間に「やった!」と声を上げるような達成感があること
- (v3 追加) 保護者が子どもの画面を見たとき、レベルと進捗が一目で分かること
- (v4 追加) アプリを開いた瞬間に「楽しそうなゲームだ」と感じる第一印象があること
- (v4 追加) ブラウザのタブで favicon を見たとき、「Make 10」のアプリだと識別できること
- (v4 追加) ホーム画面のアイコンが他のアプリと並んでも見劣りせず、子ども向けの雰囲気が伝わること
- (v4 追加) 開始画面のスタートボタンが直感的で、子どもが迷わずタップできること
- (v5 追加) Level 1 の2箇所穴埋めで、1箇所目を埋めた後に2箇所目に自然に進めること（「次はどこ?」と迷わない）
- (v5 追加) Level 2 で括弧なし限定にしたことで、子どもが試行錯誤の末に正解にたどり着ける（「括弧が必要だったのか」という不可能感がなくなる）
- (v5 追加) Level 1 → Level 2 → Level 3 の難易度の階段が以前より滑らかに感じられること
- 既存の全テストがパスし、「ふつう」モードの正解判定ロジックに変化がないこと
